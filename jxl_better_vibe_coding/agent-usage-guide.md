## 一、快速参考

| Agent | 一句话 | 触发词 | 何时不用 |
|-------|--------|--------|---------|
| `@spec-writer` | 模糊想法 → 结构化需求 | "我想要一个…但不确定怎么做" | 需求已经明确时 |
| `@modular-architect` | 大任务拆成 3-5 个独立模块 | 预计超过 200 行代码 | 单文件小改动 |
| `@fast-debugger` | 贴错误日志 → 定位 → 最小修复 | 任何报错 | 需要重构级别改动时 |
| `@knowledge-pack-generator` | 代码改动 → WHY+数据流+边界+风险 | 每次代码改动后 | 纯配置/文档修改 |
| `@knowledge-map-maintainer` | 更新项目认知地图 | 一轮开发结束后 | 单个文件小改 |
| `@microworld-builder` | 生成独立交互式教学脚本 | 复杂模块需要深度理解 | 简单工具函数 |
| `@understanding-reviewer` | PR 审查可读性（非正确性） | PR Review 时 | Bug 修复验证 |
| `@understanding-gate` | 合并前理解门槛检查 | 合并前最后一步 | 日常开发中 |

### 内置 Agent

| Agent | 用途 | 示例 |
|-------|------|------|
| `Explore` | 只读搜索，扫多个文件/目录找信息 | "帮我在整个项目里找所有调用 render() 的地方" |
| `general-purpose` | 复杂多步搜索+执行任务 | "检查这个论文代码中所有网络的输入输出维度" |
| `Plan` | 设计实现方案，识别关键文件 | 开始写代码前确认架构 |

## 二、何时用哪个（实战决策树）

### 场景 1：接到一个新需求

```
需求清晰 + <200行 → 直接写
需求清晰 + >200行 → @modular-architect 拆分 → 逐个实现
需求模糊          → @spec-writer → 确认规格 → 判断行数 → 同上
```

### 场景 2：代码写完了

```
每次代码改动           → @knowledge-pack-generator（生成知识包）
一轮开发结束           → @knowledge-map-maintainer（更新认知地图）
PR Review              → @understanding-reviewer（可读性审查）
合并前                 → @understanding-gate（理解门槛检查）
```

### 场景 3：遇到错误

```
任何报错 → @fast-debugger
  ↓
  不要自己逐行排查
  ↓
  贴完整错误日志 + 相关文件路径
  ↓
  只修复当前 bug，不顺手重构
```

### 场景 4：理解复杂代码

```
不熟悉的模块           → @microworld-builder（生成教学脚本跑一遍）
需要在多个文件搜信息    → Explore agent（比手动 grep 更系统）
需要全量代码审查        → general-purpose agent（比手动读更全面）
```

### 场景 5：项目文档

```
需要深度研究报告         → deep-research skill（多源搜索+交叉验证）
需要绘制图表             → dataviz skill（调色板+标记规范）
项目初始化               → init skill（生成 CLAUDE.md）
```

## 三、实战经验（本项目的教训）

### ✅ 有效模式

#### 模式 1：论文速览 → 代码审查 → 修正文档（三轮递进）

```
第一轮：基于论文速览做初步对比分析（docs/grad-nav-comparison.md v1）
   ↓
第二轮：审查实际代码（grad_nav-main/）发现 5 个关键纠正
   ↓
第三轮：更新对比文档 + 阅读列表 + 项目计划
```

**教训**：论文速览是不够的。代码真相往往和论文描述不同（如 SqueezeNet 冻结、观测是 57D 向量而非图像）。**论文速览后必须做代码审查，文档要留修正空间。**

#### 模式 2：交叉一致性验证

修改 5 个文件后，用 grep 做交叉引用检查：
```bash
grep -c "DDRL" file1.md file2.md file3.md  # 术语一致性
grep -l "grad-nav-comparison" *.md          # 引用链路完整
```

**教训**：多文件修改后做一次交叉验证能发现遗漏。

#### 模式 3：计划文档分层

```
MASTER_PLAN.md   ← 写死了就不会再改（步骤、时间、依赖）
PROGRESS.md      ← 活文档（状态、问题、决策、路径）
```

**教训**：计划和进度分开维护——计划是蓝图，进度是日志。合在一起会让文档不可维护。

#### 模式 4：代码审查分优先级

读一个开源项目时，按这个顺序最有效：
```
1. README.md + 顶层目录结构         → 整体架构
2. 核心算法文件 (gradnav.py)         → 训练循环
3. 环境文件 (drone_ppo.py)           → 观测/动作/reward 真相
4. 模型文件 (actor.py, vae.py)       → 网络结构真相
5. 渲染器 (gs_local.py)              → 集成方式
6. 配置文件 (yaml)                   → 超参数真相
```

**教训**：不要按目录字母序读。先算法→环境→模型→渲染→配置。

### ❌ 反模式

1. **不要一次让 Agent 做太多**——"审查整个代码库"会超时/输出被截断。一次只给一个具体任务。
2. **不要跳过代码直接信论文**——GRaD-Nav 论文说"CENet 环境编码器"，代码里是 VAE。论文说 CNN 端到端训练，代码里是冻结 SqueezeNet。
3. **不要让 Agent 同时修改多个不相关的文件**——容易产生不一致。一次 Agent 调用只做一类修改。
4. **不要在没确认的情况下删除文件**——即使是"显然无用的"。先列表确认再删。

## 四、Agent 组合工作流

### 日常开发流程

```
实现功能
  ↓
@knowledge-pack-generator  → 知识包（WHY+数据流+边界+风险+3个自测题）
  ↓
@understanding-reviewer     → 可读性审查（变量名、注释、设计决策可见性）
  ↓
修复理解性问题
  ↓
PR Ready
  ↓
@understanding-gate         → 最终理解门槛检查
  ↓
合并
```

### 论文阅读流程

```
拿到新论文
  ↓
1. 让 agent 读论文 PDF（pdftotext 提取全文）
  ↓
2. 提炼：摘要/方法/实验/局限
  ↓
3. 与自己的项目做逐项对比
  ↓
4. 更新 reading-list-v2.md
  ↓
5. 标注引用优先级 (P0/P1/P2/P3)
```

### 代码审查流程（审查别人的开源项目）

```
1. Explore agent: 扫目录结构 → 架构图
2. general-purpose agent: 读核心文件 → 技术摘要
3. 人工确认关键发现
4. 写审查报告 → 标注与自己的差异/可复用部分
```

## 五、技巧清单

| # | 技巧 | 适用场景 |
|---|------|---------|
| 1 | 给 agent 明确的**输出格式**要求（表格/JSON/bullet list） | 需要结构化结果时 |
| 2 | 先让 agent **列文件清单**再决定读哪些 | 不熟悉的代码库 |
| 3 | 长文档修改后做 **grep 交叉验证** | 多文件一致性 |
| 4 | 进度追踪用 **checkbox** 而非大段文字 | PROGRESS.md |
| 5 | 大任务**先做 100ep 微型实验**（<1h）再全量训练 | 训练任务 |
| 6 | 每个关键设计决策记录 **WHY**（不只是 WHAT） | 知识包 |
| 7 | 计划文档和进度文档**分开维护** | 项目管理 |
| 8 | 一个 Prompt = 一个模块（≤200 行） | 代码生成 |
