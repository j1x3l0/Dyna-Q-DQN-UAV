## 有效模式 ✅

> 被验证有效的做法，值得推广和复用。

### 教训: Dyna-Q 集成先验证下层 DQN 再整体训练
- **日期:** 2026-07-19
- **分类:** 有效模式 ✅
- **场景:** 首次集成 Dyna-Q 到分层框架
- **问题:** 直接运行端到端训练——上层 MADDPG + 下层 Dyna-DQN 同时学习
- **根因:** 两个学习过程耦合时，无法判断性能变化是来自上层还是下层
- **教训:** 分层 DRL 的改动应先固定上层、单独验证下层；确认下层正确后再联合训练
- **行动:** 新算法集成顺序：固定上层 → 验证下层 → 固定下层 → 验证上层 → 联合训练
- **关联:** @training-monitor, `src/hierarchical_agent.py`

### 教训: 先跑 500 轮微型实验再全量训练
- **日期:** 2026-07-11
- **分类:** 有效模式 ✅
- **场景:** 新算法验证阶段
- **问题:** 直接跑 5000 轮全量训练，结果 3 小时后发现参数错误
- **根因:** DRL 训练的反馈周期太长，小错误在长期训练中才会暴露
- **教训:** 500 轮验证（<30min）→ 确认无误 → 5000 轮正式训练
- **行动:** 任何新实验必须先跑 500ep 烟雾测试，确认 reward 曲线趋势正确
- **关联:** `scripts/run_test_training.py`, @training-monitor

### 教训: 奖励函数分解分析帮助定位训练瓶颈
- **日期:** 2026-07-14
- **分类:** 有效模式 ✅
- **场景:** 训练停滞在 reward ≈ -1000，无法判断是哪个 reward 项的问题
- **问题:** 总 reward 的移动平均看起来不错，但内部成分不均衡
- **根因:** 聚合指标掩盖了子指标的退化——能效提升但碰撞惩罚增加
- **教训:** 把 reward 分解为 throughput / energy / collision / fairness 子项单独监控
- **行动:** 训练监控应同时展示总 reward 和各成分的曲线
- **关联:** `scripts/reward_decomposition.py`, @experiment-analyzer

## 反模式 ❌

> 被验证有害的做法，必须避免。

### 教训: 不要在 GPU 服务器上直接修改运行中的代码
- **日期:** 2026-07-13
- **分类:** 反模式 ❌
- **场景:** 训练运行中修改 src/ 下的核心文件
- **问题:** 训练读入修改后的代码，行为变化但没有任何记录
- **根因:** Python 的 import 机制——运行中修改导入的模块，后续 import 读到新版本
- **教训:** 训练运行中修改代码 = 实验不可复现。必须先停训练 → 改代码 → 重启。
- **行动:** 用 screen 分离训练进程和开发环境；改代码前确认 `screen -ls` 中无训练进程
- **关联:** memory `server-connection.md`, @training-monitor

### 教训: 不要基于单次实验结果下结论
- **日期:** 2026-07-12
- **分类:** 反模式 ❌
- **场景:** 跑了一次 MADDPG 5000ep，reward 比分层差 63%，就下结论"MADDPG 不如分层"
- **问题:** 单次实验的随机种子效应可能产生误导
- **根因:** DRL 训练有高方差——同一算法不同种子可能产生 ±20% 的性能差异
- **教训:** 关键结论至少需要 3 次独立运行（不同种子）来确认
- **行动:** 基准测试默认 3 seeds；报告同时展示均值和标准差
- **关联:** @experiment-analyzer, `scripts/run_full_benchmark.py`

## 陷阱 ⚠️

> 容易踩的坑，表面看起来没问题但实际有风险。

### 教训: Dyna-K 取值不是越大越好
- **日期:** 2026-07-15
- **分类:** 陷阱 ⚠️
- **场景:** 尝试 K=50 的 Dyna-Q 规划步数
- **问题:** K 值增大后训练反而不稳定，reward 方差显著上升
- **根因:** 环境模型预测误差在 K>10 后累积放大——模型在状态空间稀疏区域预测不准
- **教训:** Dyna-K 存在最优区间（本项目约 5-10），超过后模型误差 > 真实交互收益
- **行动:** 新项目做 Dyna-K 扫描 (1, 3, 5, 10, 20)；绘制 K vs (收敛速度, 最终性能, 稳定性)
- **关联:** `src/hierarchical_agent.py::dyna_plan()`, @experiment-analyzer

## 洞察 💡

> 深层认知突破，改变了对某个问题的理解方式。

### 教训: 分层架构的主要增益来自决策解耦而非算法差异
- **日期:** 2026-07-16
- **分类:** 洞察 💡
- **场景:** 对比 MADDPG (单层) vs Hierarchical-DQN (分层) vs Hierarchical-DynaQ (分层+模型)
- **问题:** 分层+无模型的 Hier-DQN 已经比 MADDPG 好 75%，Dyna-Q 只多 0%
- **根因:** 在 500 轮训练中，决策解耦（分层）消除了动作空间的组合爆炸——这才是核心瓶颈。Dyna-Q 解决的是样本效率，在 500 轮时还不明显
- **教训:** 算法改进的边际收益递减——先解决架构瓶颈（分层），再优化学习效率（Dyna-Q）
- **行动:** 未来算法改进时，先回答"这个改进解决了什么瓶颈"——避免在已解耦的瓶颈上叠加优化
- **关联:** 论文 §4.3, `reports/项目评估与改进路线.md`
